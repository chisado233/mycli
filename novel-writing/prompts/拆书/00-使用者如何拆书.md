# 调度 Agent 如何组织长篇小说拆书

## 1. 本文件定位

本文件不是给“实际拆书模型”看的细则，而是给 **调度 agent / 编排 agent / 当前助手角色** 使用的操作手册。

调度 agent 的职责是：

1. 确认原书、拆书项目、输出目录。
2. 决定本轮拆书任务类型：批次素材拆解、文风拆解、阶段汇总、全书统一、还原写作、对比评估等。
3. 选择应传入哪些模块化 prompt。
4. 生成 `novel-writing agent run <request.json>` 所需 JSON。
5. 运行 request。
6. 检查产物是否按多文件拆分、是否覆盖目标章节、是否质量合格。
7. 必要时拆小批次重跑或调整模块。

拆书 prompt 目录：

```text
D:\agent_workspace\capability-library\mycli\novel-writing\prompts\拆书
```

当前约定：

```text
D:\agent_workspace\capability-library\mycli\novel-writing\prompts\拆书\agent   # agent 系统 prompt
D:\agent_workspace\capability-library\mycli\novel-writing\prompts\拆书\模板    # 分类模板 / 规则模板 / 自检模板
```

主运行命令：

```powershell
D:\agent_workspace\capability-library\mycli\mycli.ps1 novel-writing agent run <request.json>
```

## 1A. 调度 agent 如何实际调用拆书 agent

### 1A.1 调用方式总览

本拆书系统不是直接在聊天里粘贴 prompt，而是通过 JSON request 调用本地 runner。

基本流程：

```text
1. 选择本轮要用的 agent 系统提示词 md。
2. 选择本轮要传入的模板 md、原文章节 md、历史拆解 md。
3. 写一个 request.json。
4. 执行 mycli novel-writing agent run <request.json>。
5. 检查 run.json、输出.md 和 split_output_files 写出的多文件产物。
```

命令：

```powershell
D:\agent_workspace\capability-library\mycli\mycli.ps1 novel-writing agent run D:\path\to\request.json
```

如果 `mycli` 已在 PATH，也可以：

```powershell
mycli novel-writing agent run D:\path\to\request.json
```

### 1A.2 推荐的新写法：不写 agent，直接传 agent md

runner 支持不指定 `agent` 字段。调度 agent 只需要用 `prompt.path` 指向对应 agent 的 Markdown 系统提示词。

最小结构：

```json
{
  "task_type": "031-035章素材库分类拆书",
  "model": "MoreCode/gpt-5.5",
  "max_tokens": 70000,
  "temperature": 0.2,
  "prompt": {
    "path": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\prompts\\拆书\\agent\\03A-章节批量粗拆agent.md"
  },
  "user_prompt": "请对第031-035章进行素材库分类版高质量拆书。",
  "project": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\deconstruction\\书名",
  "base_dir": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\deconstruction\\书名",
  "target": "output\\batch-031-035\\_raw.md",
  "output_dir": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\deconstruction\\书名\\agent-runs",
  "split_output_files": true,
  "split_output_base": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\deconstruction\\书名\\output\\batch-031-035",
  "context_files": []
}
```

runner 会从 agent md 的标题自动推断 agent 名称。

### 1A.3 兼容旧写法：agent + system_prompt

旧写法仍可用：

```json
{
  "agent": "拆书-素材库分类agent",
  "task_type": "031-035章素材库分类拆书",
  "model": "MoreCode/gpt-5.5",
  "system_prompt": {
    "path": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\prompts\\拆书\\agent\\03A-章节批量粗拆agent.md"
  },
  "user_prompt": "请对第031-035章进行素材库分类版高质量拆书。",
  "context_files": []
}
```

但推荐新写法，减少 `agent` 名称和系统提示词路径不一致的问题。

### 1A.4 先 build-prompt 预览，不调用模型

正式跑模型前，可以先生成最终拼装后的 prompt 检查：

```powershell
D:\agent_workspace\capability-library\mycli\mycli.ps1 novel-writing agent build-prompt D:\path\to\request.json --out D:\path\to\preview-prompt.md
```

检查重点：

```text
- Agent 是否正确。
- System Prompt 是否读取到 agent md。
- 用户提示词是否明确目标范围。
- context_files 是否包含模板、目标章节、上下文章节。
- 目标章节和上下文章节 label 是否区分清楚。
```

### 1A.5 collect-context：只检查上下文展开

如果 request 中使用了 `relate: true` 自动关联上下文，可以先检查展开结果：

```powershell
D:\agent_workspace\capability-library\mycli\mycli.ps1 novel-writing agent collect-context D:\path\to\request.json --out D:\path\to\context-check.json
```

普通拆书通常不需要 `relate: true`，直接显式列出 context_files 更稳定。

### 1A.6 run 后检查哪里

运行完成后，runner 会输出模型原始输出路径，通常在：

```text
<output_dir>\<timestamp>\输出.md
<output_dir>\<timestamp>\提示词.md
<output_dir>\<timestamp>\run.json
```

如果 request 设置了：

```json
"split_output_files": true,
"split_output_base": "D:\\...\\output\\batch-031-035"
```

还会把模型输出中的：

```text
<<<FILE: 相对路径/文件名.md>>>
```

拆成多个实际文件，写入 `split_output_base`。

必须检查：

```text
run.json 中 ok 是否为 true。
run.json 中 split_output_files.blocks 是否大于 0。
split_output_base 下是否出现 00-总览、01-剧情内容拆解、02-可复用素材库、03-索引、04-质量自检。
目标章节是否完整覆盖。
```

### 1A.8 调度 agent 生成 request 的最低检查清单

写完 request 后，调度 agent 必须检查：

```text
- 是否有 prompt.path 或 system_prompt.path。
- prompt.path 是否指向 prompts\拆书\agent 下的正确 agent md。
- context_files 是否包含本轮必需模板。
- context_files 是否包含目标章节。
- 5章拆解时是否包含前后各约10章上下文。
- user_prompt 是否明确“只输出目标章节，不输出上下文章节”。
- split_output_files 是否为 true。
- split_output_base 是否指向本批次独立输出目录。
- target 是否指向 _raw.md。
- model 是否为 MoreCode/gpt-5.5。
```

## 2. 拆书总结构

长篇小说拆书最终分三大成果：

```text
一、可复用素材库
  场景、爽点、伏笔、人物设定、情感线推进、力量体系、世界设定、完整故事概要、文学风格、灵活记录

二、剧情内容拆解
  章节事件、因果链、场景顺序、人物目标、信息释放、伏笔推进、结尾钩子

三、小说文风拆解
  叙述节奏、吐槽机制、语言风格、对话口吻、爽点表达方式、文雅/通俗切换
```

调度 agent 不需要在一个 request 里完成所有事情。应按任务类型分批调度。

## 3. 批次粒度建议

### 3.1 高质量素材拆解

推荐：**5 章一批**。

原因：

- 高质量素材版要求大段原文摘录、YAML、写法解析、可复用变体，信息密度高。
- 10 章容易超时。
- 5 章更容易稳定输出多文件。

上下文建议：

```text
目标：5章
上下文：前后约10章
```

示例：

```text
目标：021-025
上下文：011-035
```

### 3.2 粗扫定位

如果只是判断哪些章节值得精拆，可以 20 章粗扫。

### 3.3 阶段汇总

建议每 50 章或每卷汇总一次。

### 3.4 全书统一

全书跑完后再做。

## 4. request JSON 基本字段

`novel_agent_runner.py` 支持以下关键字段：

```json
{
  "agent": "拆书-素材库分类agent",
  "task_type": "021-025章素材库分类拆书",
  "model": "MoreCode/gpt-5.5",
  "max_tokens": 70000,
  "temperature": 0.2,
  "system_prompt": {
    "path": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\prompts\\拆书\\agent\\03A-章节批量粗拆agent.md"
  },
  "user_prompt": "具体任务说明",
  "project": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\deconstruction\\书名",
  "base_dir": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\deconstruction\\书名",
  "target": "相对输出路径\\_raw.md",
  "output_dir": "D:\\...\\agent-runs",
  "split_output_files": true,
  "split_output_base": "D:\\...\\output\\batch-021-025",
  "context_files": []
}
```

字段说明：

| 字段 | 作用 |
|---|---|
| `agent` | 本轮 agent 名称，便于记录 |
| `task_type` | 任务类型，便于审计 |
| `model` | 默认用 `MoreCode/gpt-5.5` |
| `max_tokens` | 高质量素材拆解建议 50000-70000 |
| `temperature` | 拆书建议 0.2；写作还原可 0.5 左右 |
| `system_prompt.path` | 主 agent prompt |
| `user_prompt` | 本轮具体目标、范围、输出要求 |
| `project/base_dir` | 拆书项目根目录 |
| `target` | 原始模型输出保存位置，可相对 project |
| `output_dir` | agent run 记录目录 |
| `split_output_files` | 必须设为 true，启用多文件拆分 |
| `split_output_base` | 多文件输出根目录 |
| `context_files` | 传入模块、原文章节、既有拆解结果 |

补充：`agent` 字段现在可以省略。省略时，runner 会从传入的 agent Markdown 系统提示词中自动推断名称。

`system_prompt` 也可以不写，直接使用以下任一字段传入 agent 的 `.md`：

```json
{
  "prompt": {
    "path": "D:\\agent_workspace\\capability-library\\mycli\\novel-writing\\prompts\\拆书\\agent\\03A-章节批量粗拆agent.md"
  }
}
```

可用别名包括：`prompt`、`提示词`、`agent_prompt`、`agent_md`、`agent_markdown`、`md_prompt`。旧的 `system_prompt` 写法仍兼容。

## 5. 多文件输出约定

调度 agent 应在 `user_prompt` 中要求模型输出多文件块。

推荐格式：

```text
<<<FILE: 相对路径/文件名.md>>>
# 文件标题
正文
```

runner 也兼容：

```text
FILE: 相对路径/文件名.md
正文
```

但仍推荐使用 `<<<FILE: ...>>>`。

路径要求：

- 必须是相对路径。
- 不得使用绝对路径。
- 不得包含 `..`。
- 每个正式素材尽量单独一个 md。

## 6. 批次素材拆解 request 模板

用途：按 material-library 分类输出高质量素材 + 剧情内容拆解。

### 6.1 什么时候调用

这是完整拆书的主循环。只要用户要求“完整拆一本书”“继续拆后面章节”“按素材库分类拆书”，默认就使用这个步骤。

推荐每次处理：

```text
目标章节：5 章
上下文：目标前 10 章 + 目标 5 章 + 目标后 10 章
```

例子：

```text
目标章节：031-035
上下文：021-045
```

边界处理：

```text
目标 001-005：上下文 001-015
目标 006-010：上下文 001-020
目标 706-710：上下文 696-712
```

注意：上下文范围内的非目标章节只用于理解前后因果、人物状态和伏笔，不得输出为正式目标章节拆解。

### 6.2 必须使用的 agent prompt

### 6.7 user_prompt 推荐写法

```text
请对第031-035章进行素材库分类版高质量拆书。

目标范围：031-035。
上下文范围：021-045。

重要规则：
1. 只输出目标范围 031-035 的正式拆解和素材；021-030、036-045 只作为上下文参考，不得输出为正式章节拆解。
2. 必须按 material-library 分类输出多文件。
3. 每章必须有剧情内容拆解。
4. 每条高价值素材必须包含 YAML、source、source_range、tag、status、merge_policy。
5. 高价值素材必须有大段原文证据或详细场景证据、写法解析、可复用变体、禁止照搬点。
6. 人物设定、情感线、力量体系、世界设定如果只是阶段信息，要标记为“阶段素材 / 最终统一”。
7. 伏笔必须标记状态，并评估是否巧妙。
8. 最后必须输出质量自检。
9. 输出必须使用 <<<FILE: 相对路径/文件名.md>>> 多文件格式。
```

## 9A. 最终汇总审核评估 request 模板

用途：只审核最终的阶段/全书汇总结果，不对每个 5 章批次都单独跑审核。

原因：

- 5 章一拆的批次数量太多，如果每批都跑审核，成本和调度复杂度会非常高。
- 批次拆解已经要求模型输出 `04-质量自检`，日常只需要调度 agent 做轻量检查。
- 真正影响后续写作的是阶段汇总、全书统一和精选入库候选，因此审核 agent 应集中检查这些最终汇总层产物。

默认策略：

```text
批次拆解：不单独调用审核评估 agent，只检查 run.json、拆分文件、质量自检是否存在。
阶段汇总：可选审核；如果该阶段要作为后续全书统一的重要输入，建议审核。
全书统一：必须审核。
正式入库前：必须审核精选素材候选。
```

### 9A.1 什么时候调用

建议：

```text

50章阶段汇总：可选审核。若阶段汇总质量直接影响后续全书统一，建议审核。
```

## 29. 前后十章上下文过载时的摘录降载规则

记录时间：2026-05-09

问题：十章单内容请求若传入前 10 + 目标 10 + 后 10 章全文，部分范围 prompt_tokens 超过 120k，模型可能只返回 usage 无正文，导致 split blocks=0。

用户原意仍是“原文依旧要多传前后十章”，因此降载时不能直接丢掉前后文，而应改成：

1. 目标 10 章仍传全文。
2. 前 10 章和后 10 章改为“上下文摘录文件”，每章保留标题、开头片段、结尾片段和少量关键词/钩子，不传全文。
3. context_files label 仍然使用 `上文031-摘录`、`下文060-摘录` 等，明确它们仍是前后十章上下文。
4. user_prompt 必须说明：上文/下文为摘录版，只用于理解前后因果、人物状态和伏笔；正式输出仍只针对目标 10 章。
5. 若仍超限，再进一步压缩摘录；不要减少目标 10 章全文。

## 30. 十章单内容上下文降为前后五章全文
