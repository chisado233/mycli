#### 6. SSE 架构实现（2026-06-06）

**目标**：Bridge 通过 opencode serve SSE API 实现 token 级实时流转发，替代旧的 `spawn opencode --format json` 方案。

**实现的步骤与关键决策**：

##### 6a. 探索 opencode serve API

踩坑过程：
1. 最初尝试 `GET /sse`, `/stream`, `/api/sse` 等路径 → 全被 SPA 拦截返回 HTML
2. 尝试 WebSocket 升级 `/ws` → 失败（timeout）
3. 最终发现 `POST /session` 返回 JSON（非 HTML），顺着找到 API 模式
4. `POST /session/{sid}/prompt_async` 返回 `204` → 确认异步发消息端点
5. `GET /event` 返回 `text/event-stream` → 确认 SSE 流端点

**验证脚本**：`D:\agent_workspace\tmp\oc-sse-test.js`, `oc-api-test.js`, `oc-sse-full.js`

##### 6b. Bridge 改造（`local-agent.ts`）

**关键改动**：

| 改动 | 位置 | 说明 |
|------|------|------|
| 自动拉起 serve | `ensureOpenCodeServe()` | bridge 启动时 spawn `opencode serve --port 4096 --hostname 127.0.0.1`，30 次重试等待 |
| SSE 连接 | `connectSSE()` | `http.get(/event)` → 解析 `data:` 行 → 原样 JSON 转发 |
| 发消息 | `sendPromptAsync(sid, text)` | POST `{ parts: [{ type: "text", text }] }` → 204 |
| 创建 session | `createSession()` | POST `/session` 空 body → 获取新 session ID |
| 消息处理 | `handleBridgeMessage()` | 从 session store 获取或创建新 session → `sendPromptAsync` |
| 事件透传 | SSE handler 内的 `sendBridge()` | `{ type: "bridge.sse", conversationId, eventType, data }` — 原样 JSON |

**SSE 解析关键点**：
- opencode serve 的 SSE 格式只有 `data:` 行，**没有 `event:` 行**
- 事件类型在 JSON 的 `type` 字段里（如 `message.part.delta`）
- **绝对不能**按 `event:` 行过滤，否则所有事件被丢弃（踩坑记录）

##### 6e. Session 管理关键发现

- `POST /session/{oldSid}` 返回 **SPA HTML**（被拦截），不能用
- 必须用 `POST /session`（无路径参数）创建新 session → 返回 `{ id: "ses_xxx" }`
- agent-cli 存的旧 session ID 对 serve 无效，必须清除 `agent-cli-sessions.json`
- 首次消息自动 `createSession()` → 后续消息复用

##### 6h. 踩过的坑

1. **`--format json` 不输出 text** — 只有 step_start/tool_use/step_finish
2. **SSE 只有 data: 没有 event:** — 不能用 `event:` 行过滤解析
3. **旧 session 对 serve 无效** — 必须 `POST /session` 创建新 session
4. **`POST /session/{sid}` 返回 HTML** — SPA 拦截，无效端点
5. **不要杀 opencode 进程** — serve 进程（PID 59420）需手动管理
6. **两个源码副本** — `projects\chat_soft`（文档）vs `capability-library\mycli\channels\chat-soft\source`（实际运行）
