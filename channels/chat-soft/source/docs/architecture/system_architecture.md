# Chat Soft 总体架构设计

## 1. 架构结论

V1 采用“小服务器中心化 + 客户端本地存储 + 电脑端本地开放接口”的架构。

这是当前最适合该项目的方案，因为它同时满足：

- 跨网络稳定性
- 离线补收
- 多设备同步
- 历史消息服务端保留
- AI 调用电脑端能力
- 后续演进到文件与实时音视频

## 2. 系统组成

### 2.1 Android 客户端

职责：

- 文本消息发送与接收
- 语音消息录制与播放
- 本地数据库存储
- 推送/本地通知
- 与服务端保持长连接

### 2.2 Windows 客户端

职责：

- 文本消息发送与接收
- 语音消息录制与播放
- 本地数据库存储
- 桌面通知
- 本地开放接口
- 与服务端保持长连接

### 2.3 Chat Server

职责：

- 设备鉴权
- 会话路由
- 文本消息转发
- 离线消息缓存与补发
- 消息状态同步
- 多设备消息同步
- 媒体元数据管理

### 2.4 Media Storage

职责：

- 存放语音消息文件
- 后续扩展图片、视频、文件

V1 可以先和 Chat Server 共机部署。

### 2.5 Desktop Local Agent

部署在 Windows 客户端本机。

职责：

- 向本地 AI 暴露 HTTP API
- 向本地 AI 暴露 WebSocket
- 提供 CLI
- 与桌面主程序共享消息能力
- 做权限验证和本机访问控制

## 4. 为什么 V1 不做纯点对点

纯点对点的主要问题：

- NAT 穿透不稳定
- 离线补收困难
- 消息历史服务端保留困难
- 多设备同步复杂
- AI 自动化场景不够稳定

因此 V1 先采用服务端中心化消息链路。

## 5. 核心通信流程

### 5.1 文本消息流程

1. 发送端本地落库，状态为 `sending`
2. 客户端通过长连接把消息发给 Chat Server
3. 服务端写入消息表
4. 服务端投递给目标设备
5. 目标设备收到并本地落库
6. 服务端回传送达状态
7. 目标设备阅读后回传已读状态

### 5.2 语音消息流程

1. 发送端录制音频
2. 本地保存临时音频文件
3. 上传音频到服务端媒体存储
4. 服务端返回媒体标识
5. 客户端发送语音消息元数据
6. 接收端收到后下载或按需拉取音频
7. 接收端本地缓存并播放

## 6. 多设备同步策略

V1 采用“服务端为同步中枢”的方案：

- 每台设备拥有独立 `device_id`
- 同一用户下可有多台设备
- 消息在服务端保留
- 设备上线后按游标补拉

## 7. 本地开放接口设计原则

### 7.1 接口形式

- HTTP：用于命令式调用
- WebSocket：用于实时事件订阅
- CLI：用于脚本化操作

### 7.2 V1 最小能力

- 发送文本消息
- 读取最近消息
- 发送语音消息
- 搜索本地历史

### 7.3 安全边界

- 仅监听 `127.0.0.1`
- 必须携带本地 token
- 默认关闭远程访问
- 后续支持显式授权机制

## 8. 存储设计

### 8.1 客户端本地存储

- SQLite
- 存储消息、会话、设备信息、媒体索引、同步游标

### 8.2 服务端存储

- PostgreSQL
- 存储设备、会话、消息、消息状态、同步游标、媒体元数据

### 8.3 媒体存储

- V1 可先使用本机磁盘目录
- 后续可扩展对象存储

## 9. 安全设计原则

- 所有外部连接走 HTTPS/WSS
- 设备首次绑定使用配对密钥
- 后续使用短期访问令牌 + 刷新机制
- 预留消息内容端到端加密扩展位
- 本地开放接口必须本机回环访问

## 10. 演进路线

### V1

- 中心化文本消息
- 中心化语音消息
- 本地 API

### V2

- 媒体消息与文件能力增强
- 更完整同步与搜索

### V3

- WebRTC 1 对 1 实时语音/视频
- STUN/TURN 或中继支持
