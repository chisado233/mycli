# Codex 镜像需求沟通记录

更新时间：2026-04-17

## 目标

围绕“Codex 镜像”能力，逐条澄清需求，并沉淀为后续设计与开发依据。

## 当前已确认背景

- Codex 镜像的目标对象是 VS Code 里的 Codex 插件。
- 目标是同步 Codex 插件中的多个会话，并保持不同会话消息独立。
- 需要支持双向同步：
  - 在 Codex 插件里发送消息后，镜像侧可以同步看到。
  - Codex 回复消息后，镜像侧可以同步看到。
  - 外部对镜像中的某个会话发送消息后，消息需要进入 VS Code 的 Codex 插件对应会话。
- 需要支持的控制能力：
  - 切换会话
  - 切换模型
- 需要对外接口：
  - 查询每个会话及其消息
  - 向指定会话发送消息
- 当前阶段只给用户本人使用。
- 当前阶段先不优先处理手机端，先在本地把链路跑通。
- 后续架构需要方便对接手机端。

## 沟通方式

- 采用“一次只问一个问题”的方式推进。
- 每一轮问题的回答都记录在本文档中。

## 问题记录

### Q1

会话的唯一标识是否直接沿用 Codex 插件内部的 thread/conversation id，而不是由镜像层重新生成一套 id？

### A1

- 同意直接沿用 VS Code 里的 Codex 会话内部 id 作为镜像层的唯一会话标识。
- 额外要求：
  - 镜像层需要支持对会话进行“本地重命名”。
- 这个重命名只作用于镜像层显示与接口返回，不需要回写到 Codex 插件。
- Codex 插件中的原始会话标题不做修改。

### Q2

“同步所有会话”的范围是：

- 当前工作区会话
- 当前机器上的所有会话
- 或者仅镜像曾经同步过的会话

### A2

- 目标范围尽可能大。
- 希望覆盖 Codex 插件中的所有历史会话，而不只是当前工作区或当前活跃会话。
- 这意味着镜像层需要尽可能发现并同步历史会话，并提供统一访问能力。
- 如果底层实现存在限制，需要优先选择“可覆盖更多历史会话”的方案。

### Q3

模型切换希望支持到哪一层：

- 仅已有会话切换
- 仅默认模型
- 或两者都支持

### A3

- 选择“两者都支持”。
- 具体包括：
  - 支持给某个已有会话切换模型。
  - 支持设置镜像层的新会话默认模型。

### Q4

镜像对外接口形式希望是：

- 仅 HTTP REST
- 仅 WebSocket
- 或 REST + WebSocket

### A4

- 期望接口能力尽可能全面。
- 当前倾向于同时提供：
  - HTTP REST：用于查询、发送、切换、管理类操作。
  - WebSocket：用于消息与状态的实时推送。
- 如果后续还需要，也可以继续扩展其他接口形式，但第一优先级是 REST + WebSocket。

### Q5

同步实时性的期望是：

- 1 到 2 秒级
- 几百毫秒级接近实时
- 或只要求最终一致

### A5

- 期望尽可能快。
- 需要支持流式同步，因为 Codex 的输出本身是流式生成的。
- 这意味着镜像层不仅要同步“最终消息”，还要尽量同步生成中的增量内容。

### Q6

镜像保存和对外提供的消息结构，希望保留到什么程度：

- 仅纯文本
- 中等结构化信息
- 或尽量保留 Codex 原始结构

### A6

- 选择尽量完整保留原结构。
- 希望镜像层保留 Codex 消息与事件的原始结构，而不是只抽取文本。
- 这包括后续可能存在的：
  - 用户消息
  - assistant 消息
  - 流式增量片段
  - 工具调用
  - 中间事件
  - 最终完成内容

### Q7

当外部系统向镜像中的某个会话发消息时，可接受的实现方式是：

- 只接受 Codex 插件内部/官方接口
- 先走内部接口，不行再做本地 UI 模拟兜底
- 只要能稳定发进去就行

### A7

- 优先走 Codex 插件内部/官方接口。
- 如果无法完全通过内部接口实现，可以接受本地 UI 模拟作为兜底。
- 核心目标是“消息最终能稳定进入目标会话”。

### Q8

当镜像层切换当前会话时，是否需要同步切换 VS Code 中可见的 Codex 当前会话？

### A8

- 选择需要同步切换。
- 即镜像层中的当前会话切换后，VS Code 中可见的 Codex 当前会话也要跟着切换到对应会话。

### Q9

镜像层是否需要支持“新建会话”能力？

### A9

- 需要支持新建会话。
- 期望镜像接口可以：
  - 新建一个 Codex 会话
  - 可选指定模型
  - 返回新会话 id
  - 后续允许继续向该新会话发送消息

### Q10

是否接受镜像层维护本地数据库 / 本地持久化缓存？

### A10

- 接受本地数据库或本地持久化缓存方案。
- 该持久化层可用于：
  - 缓存会话列表
  - 缓存消息与流式片段
  - 保存镜像侧会话重命名
  - 保存模型设置
  - 断点恢复
  - 为后续手机端与外部接口提供稳定数据源

### Q11

Codex 镜像第一阶段希望以什么形态落地：

- 仅本地后台服务
- 本地后台服务 + 简单调试页面
- 完整桌面应用

### A11

- 第一阶段选择“本地后台服务 + 简单调试页面”。
- 目标是先把链路跑通，并提供一个可直接查看会话、消息和发送消息的调试界面。

### Q12

调试页面第一版最希望包含哪些核心区域与布局？

### A12

- 页面布局本身不是重点。
- 第一版应以功能完整和调试效率优先，不需要在界面排版上投入过多精力。
- UI 只需要足够支撑：
  - 查看会话
  - 查看消息
  - 发送消息
  - 进行基础控制与调试

### Q13

镜像对外返回流事件时，希望采用哪种风格：

- 仅返回镜像层整理后的标准事件
- 同时返回标准事件与尽量贴近 Codex 原始结构的 raw event

### A13

- 选择同时返回两层事件。
- 需要同时提供：
  - 镜像层整理后的标准事件
  - 尽量贴近 Codex 原始结构的 raw event
- 这样后续外部系统可以按需选择高层接口或底层原始数据。

### Q14

删除会话能力希望如何处理：

- 仅删除镜像侧缓存
- 同时尝试删除 Codex 插件原会话
- 第一版不提供删除能力

### A14

- 选择仅删除镜像侧缓存。
- 即删除操作只作用于镜像层的数据与展示，不修改 Codex 插件中的原始会话。

### Q15

镜像启动时，如果本地已有历史数据，但 VS Code / Codex 当前未连接，希望如何表现？

### A15

- 需要明确显示 “Codex 未连接” 状态。
- 连接状态应作为镜像层的重要状态信息暴露出来，便于界面和外部接口判断当前是否在线同步。

### Q16

当 Codex 未连接时，镜像层应允许哪些操作？

### A16

- 倾向于严格限制。
- 在 Codex 未连接时，最好不允许进行实际操作，尤其是不允许发送消息。
- 即未连接状态下应以“只读/不可操作”甚至“不可用”模式为主，而不是排队补发。

### Q17

镜像中的会话列表排序希望优先按什么规则？

### A17

- 选择按 Codex 原会话最近活跃时间排序。
- 即镜像层的默认会话列表排序应尽量贴近 Codex 原始会话活跃顺序。

### Q18

镜像接口在“查消息”时，默认返回什么粒度？

### A18

- 选择默认返回整理后的消息视图。
- 原始事件 / 原始结构通过额外参数或额外接口显式获取。
- 即默认查询接口优先保证易用性，底层原始数据按需展开。

### Q19

镜像接口中的“发送消息”默认希望采用什么语义？

### A19

- 选择“两者都支持”。
- 期望同时支持：
  - 异步提交模式
  - 等待提交成功的同步模式
- 默认更倾向异步，但应允许通过参数要求“等待提交成功”后再返回。

### Q20

镜像中的会话元数据除了重命名外，是否还需要备注 / 标签 / 收藏等能力？

### A20

- 当前阶段仅需要重命名。
- 备注、标签、收藏等附加会话元数据能力暂不纳入第一阶段范围。

### Q21

镜像中的模型切换，期望如何生效？

### A21

- 选择“两者都做”。
- 具体期望：
  - 先更新镜像层记录中的当前模型设置。
  - 再尽量立即同步到 VS Code / Codex 当前会话。
- 如果立即同步存在限制，至少要保证后续发送消息时按最新模型配置生效。

### Q22

会话列表中是否需要展示连接状态、同步状态、当前模型、最后同步时间等附加状态信息？

### A22

- 当前阶段不需要在会话列表中展示这些附加状态信息。
- 会话列表应保持尽量简洁，不额外堆叠无明显价值的状态字段。

### Q23

消息列表中，Codex 的流式输出应如何展示？

### A23

- 选择“两者都要”。
- 期望同时支持：
  - 默认显示正在生成的完整消息，并在流式过程中持续更新
  - 在需要调试时，可以展开查看流式增量片段

### Q24

工具调用 / 中间事件在第一版中应如何展示？

### A24

- 工具调用与中间事件默认折叠显示。
- 需要支持手动展开查看详细内容。
- 目标是在不牺牲原始信息保留度的前提下，避免消息界面过于杂乱。

### Q25

“消息发送成功”的判定点应放在哪？

### A25

- 选择以“Codex 会话里真实出现该用户消息”为发送成功判定。
- 即仅仅提交到镜像层或桥接层还不算最终成功，必须以目标会话中确认出现为准。

### Q26

第一版是否需要支持“失败消息重试发送”？

### A26

- 需要支持失败消息重试发送。
- 即当某条消息未成功进入目标 Codex 会话时，镜像层需要提供重新发送该消息的能力。

### Q27

会话切换与发送消息之间，是否需要严格串行一致性，避免串会话？

### A27

- 必须严格保证。
- 即当用户切换会话后再发送消息时，系统需要确保消息只进入最终确认选中的那个会话。
- 宁可为此增加一些控制逻辑，也不能出现消息串到错误会话的问题。

### Q28

第一版是否需要支持“搜索会话 / 搜索消息”能力？

### A28

- 第一版不需要支持搜索会话或搜索消息。
- 当前优先级应放在同步、收发、会话切换、模型切换和流式能力跑通上。

### Q29

镜像“新建会话”时，是否需要支持初始标题 / 初始镜像显示名参数？

### A29

- 当前阶段不需要。
- 新建会话时不必额外传入初始标题，后续如有需要可再通过重命名能力修改镜像侧显示名。

### Q30

第一版是否需要支持“导出会话数据”能力？

### A30

- 需要支持导出能力。
- 期望尽可能支持多种导出格式。
- 后续优先考虑的导出方向包括：
  - JSON
  - Markdown
  - 原始事件日志 / raw event dump
