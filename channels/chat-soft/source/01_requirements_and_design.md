# Chat Soft 项目需求与设计

## 1. 项目目标

目标是从零设计一套“安卓端 + 电脑端”的通信软件，支持手机与电脑之间跨网络通信，并允许后续持续二次开发。

当前已知目标：

- 支持安卓客户端
- 支持电脑客户端
- 支持跨网络通信，不限于局域网
- 支持文本消息
- 支持语音消息
- 支持图片消息
- 支持视频消息
- 支持文件传输
- 需要具备后续自己调整和扩展的空间

## 2. 需求访谈记录

### 第 1 轮待确认问题

### 第 1 轮确认结果

- 产品定位：
  - 仅供你自己使用
  - 暂不考虑泛用型陌生人社交或大规模多人场景
- 电脑端平台：
  - 仅 Windows
- 手机端平台：
  - 华为 P60
  - 安卓系统
- 部署条件：
  - 如需内网穿透，可以租一台小服务器做桥接
- 账号体系：
  - 不需要注册、登录、找回密码等账号系统
- 功能期望：
  - 文件传输
  - 实时语音
  - 实时视频
  - 普通音频发送
  - 普通视频发送

### 当前理解

当前更像是在做一套“个人双端私有通信系统”，重点不是社交关系链，而是：

- 自己的手机和自己的 Windows 电脑之间稳定互通
- 可跨网络
- 具备消息、文件和实时音视频能力
- 后续可以自己继续改造

### 第 2 轮待确认问题

### 第 2 轮确认结果

- 设备规模：
  - 未来可能扩展到多台设备
- 历史消息：
  - 需要长期保存
  - 需要可回看
- 文件传输：
  - 目标支持约 1GB
  - 需要考虑断点续传
- 实时音视频：
  - 优先低延迟
- 通知能力：
  - 需要
- 服务器策略：
  - 可租可不租
  - 如果不租会明显影响稳定性，则接受租用小服务器
- 消息状态：
  - 需要发送状态与失败状态
  - 需要已读等状态

### 当前产品轮廓

当前产品更接近：

- 面向个人私有使用的多设备通信系统
- 以“自己的手机 / 自己的 Windows 电脑 / 后续自己的其他设备”为主
- 具备本地消息历史、文件传输、实时音视频、通知和状态同步
- 可以接受适度的服务端参与，以换取稳定性和跨网络可用性

### 第 3 轮待确认问题

### 第 3 轮确认结果

- 历史消息存储：
  - 本地和服务器都保存
- 语音消息体验：
  - 按住录音发送
  - 播放进度条
- 视频消息体验：
  - 自动压缩
  - 原视频发送
  - 自动生成缩略图
- 实时音视频：
  - 暂时只考虑 1 对 1
- 安全：
  - 由方案设计阶段统一把控
  - 不先口头限定到某一种实现，但需要纳入核心设计
- 电脑端开放接口：
  - 必须预留给 AI 使用的开放接口
  - 这是桌面端设计的硬需求

### 新增关键约束

- 电脑端不是一个纯 GUI 客户端
- 电脑端需要同时具备“被 AI 调用”的能力
- 这意味着桌面端后续至少要考虑：
  - 本地 API
  - 自动化控制接口
  - 权限控制
  - 消息、文件、通话能力的可编排性

### 第 4 轮待确认问题

### 第 4 轮确认结果

- 多设备同步：
  - 都要
  - 所有设备自动同步同一份会话历史
- 离线补收：
  - 都要
  - 设备恢复在线后应自动补收离线期间消息
- 版本策略：
  - 接受分阶段实施
  - 第一版尽量简单
  - 先把最基础的文本消息和语音消息跑通
  - 先观察实际延迟与稳定性
  - 后续再补齐其他功能
- 电脑端开放接口：
  - 本地 HTTP API
  - 本地 WebSocket
  - 命令行接口
  - 三者都要
- AI 可调用能力：
  - 原则上都要
  - 发送、读取、搜索、文件、通话、联系人、设备管理都应纳入长期规划

## 3. 初步范围划分

### V1 最小可用版本

目标：

- 尽快验证跨网络消息链路
- 尽快验证语音消息链路
- 测量延迟、稳定性与基本可用性

范围：

- 安卓端
- Windows 端
- 1 对 1 通信
- 文本消息
- 语音消息
- 消息历史保存
- 本地通知
- 服务端消息持久化
- 多设备基础同步能力
- 电脑端基础开放接口

暂缓到后续版本：

- 实时语音通话
- 实时视频通话
- 图片消息
- 普通视频消息
- 大文件传输
- 断点续传
- 更完整的设备管理
- 更完整的 AI 调用能力

### V2 扩展版本

- 图片消息
- 普通音频/视频文件发送
- 文件传输能力增强
- 断点续传
- 更完整的搜索与同步
- 电脑端 API 扩展

### V3 增强版本

- 实时语音通话
- 实时视频通话
- 更强的弱网与重连优化
- 更完整的多设备体验
