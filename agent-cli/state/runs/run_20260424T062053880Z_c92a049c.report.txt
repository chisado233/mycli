# Project Manager 方案草案

## 1. 核心定位

`project-manager` 不应该重复 `agent-workflow` 的功能。

我建议它定位为：

> **面向长期项目的目标、需求、任务、决策、上下文和交付状态管理层。**

它负责回答：

- 这个项目是什么？
- 当前目标是什么？
- 有哪些需求、任务、风险、决策？
- 哪些任务已经完成，哪些正在进行，哪些被阻塞？
- 当前 agent 应该接着做什么？
- 之前为什么这么做？
- 项目材料、运行产物、讨论记录和决策记录在哪里？
- 什么时候应该启动一个 workflow，什么时候只需要一次 agent run？

而 `agent-workflow` 负责回答：

- 给定一个明确流程，怎么按节点执行？
- Step 怎么调度？
- Artifact 怎么保存？
- Review 怎么挂起/恢复？
- Listener 怎么等待事件？

所以二者关系应是：

```text
project-manager = 项目治理 / 任务治理 / 上下文治理 / 交付治理
agent-workflow = 流程执行引擎 / Step runtime
agent-cli      = 单次 agent 调用入口
```

---

## 2. 为什么需要 project-manager

当前 workspace 已经有：

- `mycli agent-cli`：单次 agent 调用
- `mycli agent-workflow`：长期或固定流程执行
- `skill-library`：能力检索与注册
- `agent-system-rules`：规则和流程层能力

但还缺一个“项目层大脑”。

典型问题是：

1. 用户说“帮我做某某项目”，agent 往往只有当前会话上下文。
2. 多轮工作后，需求、决策、未完成事项容易散落在聊天里。
3. `todo` 只是会话内任务清单，不是项目级状态。
4. `agent-workflow` 适合固定流程，但很多项目是开放式、演进式的。
5. 多 agent 协作时，需要有统一的项目状态和任务边界。
6. 每次继续项目时，需要快速恢复上下文，而不是重新读一堆材料。

因此 `project-manager` 应作为：

> **长期项目的状态账本 + 决策日志 + 任务看板 + 上下文入口 + workflow 调度前置层。**

---

## 3. 与 agent-workflow 的边界

### 3.1 Project Manager 管“项目”

它管理：

- Project metadata
- Goals
- Scope
- Milestones
- Requirements
- Task backlog
- Current active task
- Decisions
- Risks
- Constraints
- Context files
- Delivery artifacts
- Review records
- Handoff summaries

### 3.2 Agent Workflow 管“流程实例”

它管理：

- Workflow definition
- Step definition
- WorkflowRun
- StepRun
- RuntimeEvent
- Artifact Store
- Listener/Approval/Retry/Goto
- DebugRun

### 3.3 调用关系

建议是：

```text
project-manager
  ├─ 可创建/引用 workflow project
  ├─ 可启动 agent-workflow run
  ├─ 可记录 workflow run_id 到 project task
  ├─ 可把 workflow 输出纳入 project artifact
  └─ 可根据项目状态决定下一步动作

agent-workflow
  └─ 不反向依赖 project-manager
```

也就是说：

- `agent-workflow` 保持底层执行引擎纯净。
- `project-manager` 作为更上层的“项目治理器”使用 workflow。

---

## 4. 适用场景

使用 `project-manager` 当任务具有以下特征：

1. 需要跨会话持续推进。
2. 有多个需求、任务、阶段或里程碑。
3. 有需求澄清、方案讨论、设计决策、实现、验证、复盘等阶段。
4. 需要记录“为什么这么做”。
5. 需要保存项目材料和交付物。
6. 需要多人/多 agent 协作。
7. 任务不是固定 workflow，但可能在局部调用 workflow。
8. 用户说：
   - “帮我管理这个项目”
   - “接着上次那个项目”
   - “给这个项目建档”
   - “规划一下项目路线”
   - “把当前进展整理成项目状态”
   - “后面你要记得这个项目怎么推进”

不适合使用 `project-manager` 的情况：

- 单次问答。
- 简单代码修改。
- 一次性命令执行。
- 已经明确是一个固定 workflow 的运行任务。
- 没有长期状态价值的临时实验。

## 6. 项目实例目录结构

每个被管理项目可以有一个 `.agent-project/` 目录。

例如：

```text
some-project/
  .agent-project/
    project.json
    status.md
    backlog.json
    tasks/
      task-0001.json
      task-0002.json
    decisions/
      ADR-0001.md
      ADR-0002.md
    risks.json
    constraints.md
    context/
      index.json
      pinned-files.md
      glossary.md
      architecture.md
    artifacts/
      index.json
    runs/
      workflow-runs.json
      agent-runs.json
    handoffs/
      2026-04-24.md
    reviews/
      review-0001.md
```

其中：

- `project.json`：结构化项目事实来源。
- `status.md`：人类可读状态页。
- `backlog.json`：任务队列。
- `tasks/`：每个任务一个结构化文件。
- `decisions/`：重要决策记录，类似 ADR。
- `risks.json`：风险、阻塞项和待确认事项。
- `constraints.md`：约束和边界。
- `context/`：长期上下文索引。
- `artifacts/`：交付物索引，不一定存放原文件。
- `runs/`：关联的 agent/workflow 执行记录。
- `handoffs/`：跨会话交接摘要。
- `reviews/`：项目级审核和复盘。

如果不想污染项目根目录，也可以支持集中式存储：

```text
D:\agent_workspace\projects\.agent-projects\{project_id}\
```

但我更推荐默认使用项目内 `.agent-project/`，因为上下文就近、便于版本管理。

## 9. Decision 设计

项目管理里最重要的不是任务，而是决策。

建议采用轻量 ADR：

```text
decisions/
  ADR-0001-project-manager-boundary.md
  ADR-0002-storage-location.md
```

模板：

```md
# ADR-0001: Project Manager boundary

## Status

Proposed

## Context

We already have agent-workflow as a step runtime.

## Decision

Project Manager owns project state, tasks, decisions, context and delivery tracking.
Agent Workflow owns workflow execution.

## Consequences

- Project Manager may call Agent Workflow.
- Agent Workflow should not depend on Project Manager.
- Project tasks can reference workflow runs.
```

决策状态：

```text
proposed | accepted | superseded | rejected
```

## 13. Agent 使用规则

`SKILL.md` 里应明确写：

### Use This When

- 用户要求管理一个长期项目。
- 用户要求恢复项目上下文。
- 当前任务涉及多个阶段、任务、决策或交付物。
- 需要记录任务状态、需求和决策。
- 需要把 workflow run 绑定到项目任务。
- 需要生成 handoff。

### Do Not Use This When

- 一次性小任务。
- 单次问答。
- 简单代码编辑。
- 已有明确 workflow 且不需要项目级状态。

### Recommended Flow

```text
1. Locate or initialize .agent-project/
2. Read project.json
3. Read status.md
4. Read active task
5. Read relevant decisions/context index
6. Propose next action
7. Execute work
8. Update task/status/decision/handoff as needed
```

   - `handoff create`

MVP 应该先坚持“文件系统即项目状态”。

## 18. 我建议的关键设计原则

1. **项目状态优先于聊天上下文。**  
   聊天会丢，项目状态应该留。

2. **文件优先，服务后置。**  
   第一版不要上数据库或 daemon。

3. **Project Manager 不做 Step Runtime。**  
   不重造 `agent-workflow`。

4. **任务、决策、上下文分离。**  
   不要把所有东西写进一个 `memory.md`。

5. **每个重要改变都能解释“为什么”。**  
   用 Decision Records 解决这个问题。

6. **对 agent 友好，也对人类友好。**  
   JSON 负责结构化，Markdown 负责阅读。

7. **可以手工修复。**  
   文件结构应足够透明，不依赖黑盒状态。

8. **默认最小可用，不做企业级项目管理系统。**  
   它是 agent 项目管理层，不是 Jira。

## 19. 需要讨论的关键问题

我建议我们重点讨论这些：

1. `.agent-project/` 是否放在项目内？还是集中放到 workspace 某个目录？
2. Project Manager 是否要强制每个项目有 `project.json`，还是允许只有 `status.md` 的轻量模式？
3. Task 是否需要支持依赖关系？例如 `blocked_by` / `depends_on`。
4. 是否需要 Milestone？
5. 是否要从第一版就接入 `agent-workflow`，还是先只记录链接？
6. 是否要支持“自动根据状态推荐下一步”？
7. `handoff` 是每次任务完成都生成，还是用户请求/会话结束时生成？
8. 是否需要区分：
   - coding project
   - research project
   - capability package
   - workflow project
9. Project Manager 的 CLI 是用 Python 写，还是 PowerShell 足够？
10. 是否应和 `mycli agent-cli` 的 session 记录建立关联？

## 20. 我的倾向性建议

如果让我拍第一版，我会这样定：

1. 使用项目内 `.agent-project/`。
2. 第一版必须有 `project.json` 和 `status.md`。
3. Task 支持 `depends_on`，但不做复杂调度。
4. 支持 Milestone，但可选。
5. 第一版只“记录 workflow link / run_id”，暂不做深度编排。
6. 支持 `summary` 命令，为 agent 恢复上下文生成短摘要。
7. `handoff` 用户请求或阶段结束时生成，不强制每次任务都生成。
8. CLI 用 Python 写，PowerShell 只做 wrapper。
9. 注册方式模仿 `agent-workflow`：
   - 源文件在 `agent-system-rules/project-manager`
   - `registration/mycli` 同步到 `mycli/project-manager`
   - `SKILL.md` 同步到 `skill-library/project-manager`
10. MVP 先实现文件系统状态管理，不做服务化。

这套设计和现有 `agent-workflow` 的风格能对齐，同时又不会重叠。
