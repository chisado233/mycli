千束，我先把“项目状态”单独抽出来设计，不急着进入 CLI 和文件实现。

我认为这里最重要的一点是：**项目状态不能只有一个 `status` 字段**。  
如果只写：

```json
"status": "active"
```

后面一定会混乱，因为它同时想表达：

- 项目是否已经注册？
- 项目生命周期在哪个阶段？
- 当前有没有人在推进？
- 是否阻塞？
- 是否健康？
- 当前主要推进项是什么？
- 是否需要用户决策？
- 是否已完成但仍需维护？

所以我建议把项目状态拆成几个正交维度。

---

# Project Manager：项目状态设计草案

## 1. 总体原则

项目状态应该服务于四类问题：

1. **查询类**
   - 有哪些项目？
   - 哪些项目正在推进？
   - 哪些项目被阻塞？
   - 哪些项目需要我决策？
   - 哪些项目长期没更新？

2. **恢复上下文类**
   - 这个项目现在做到哪了？
   - 当前活跃任务是什么？
   - 下一步应该做什么？
   - 最近的重要决策是什么？

3. **调度类**
   - 这个项目是否允许继续推进？
   - 是否需要先确认需求？
   - 是否可以启动 agent / workflow？
   - 是否应该进入 review / handoff / archive？

4. **治理类**
   - 项目是否过期？
   - 是否状态和实际文件不一致？
   - 是否有阻塞、风险、悬空任务？
   - 是否需要维护、复盘或归档？

---

# 2. 不要单状态，要状态分层

建议项目状态分成 7 个主要维度：

```text
Project Status =
  registration_status   注册状态
  lifecycle_status      生命周期状态
  phase                 当前阶段
  activity_status       当前活动状态
  health_status         健康状态
  delivery_status       交付状态
  attention_status      是否需要关注
```

也就是说，一个项目可以同时是：

```text
registered + active + implementation + blocked + yellow + partial + needs_user
```

这比单独一个 `status = blocked` 更有表达力。

---

# 3. 项目类型 `project_type`

在状态之前，先要有项目类型。因为不同类型项目的状态含义不完全一样。

建议第一版支持这些类型：

```text
capability
workflow
coding
research
documentation
ops
experiment
external_repo
meta
```

## 3.1 类型说明

| 类型 | 含义 | 示例 |
|---|---|---|
| `capability` | 能力图书馆中的能力包 | `project-manager`, `agent-workflow`, skill |
| `workflow` | 一个 workflow 工程 | 某个 `workflow.json` 驱动的长期流程 |
| `coding` | 普通代码项目 | 一个 app、library、CLI |
| `research` | 调研型项目 | 模型对比、技术选型、资料汇总 |
| `documentation` | 文档项目 | 手册、规范、知识库 |
| `ops` | 运维/环境项目 | Clash、代理、部署、本地环境维护 |
| `experiment` | 临时实验 | tmp 下的小试验 |
| `external_repo` | 外部仓库镜像或第三方项目 | `projects/` 下 clone 的 repo |
| `meta` | agent 系统自身建设 | agent rules、system prompt、mycli 体系 |

对于当前这个 `project-manager`，我会标成：

```json
"project_type": "capability"
```

或者更细一点：

```json
"project_type": "meta"
"subtype": "agent-system-rule"
```

但第一版可以先简单用：

```json
"type": "capability"
"domain": "agent-system-rules"
```

# 4. 注册状态 `registration_status`

这个表示项目是否被 Project Manager 纳入管理。

建议状态：

```text
unregistered
discovered
registered
indexed
stale
missing
retired
```

## 4.1 状态定义

| 状态 | 含义 |
|---|---|
| `unregistered` | 尚未纳入 project-manager |
| `discovered` | 被扫描发现，但还没正式建档 |
| `registered` | 已有 `.agent-project/project.json` 或中心 registry 记录 |
| `indexed` | 已注册，并且已建立可查询索引 |
| `stale` | 项目存在，但状态长时间未刷新，或索引落后 |
| `missing` | registry 中有记录，但项目路径不存在 |
| `retired` | 项目曾被管理，但现在不再活跃管理 |

第一版可以简化为：

```text
registered
stale
missing
retired
```

但我建议设计上保留 `discovered`，因为后面做 workspace 扫描时很有用。

# 5. 生命周期状态 `lifecycle_status`

这是项目最核心的生命周期。

建议状态：

```text
idea
proposed
planned
active
on_hold
completed
maintenance
archived
cancelled
rejected
```

## 5.1 状态定义

| 状态 | 含义 |
|---|---|
| `idea` | 只是一个想法，尚未承诺推进 |
| `proposed` | 已提出，等待是否立项 |
| `planned` | 已决定做，正在规划范围/任务 |
| `active` | 正在实际推进 |
| `on_hold` | 暂停，不主动推进，但未结束 |
| `completed` | 主要目标已完成 |
| `maintenance` | 已交付，进入维护/小修状态 |
| `archived` | 已归档，默认不再推进 |
| `cancelled` | 中途取消 |
| `rejected` | 提案被拒绝，不进入执行 |

## 5.2 推荐主路径

```text
idea
  -> proposed
  -> planned
  -> active
  -> completed
  -> maintenance
  -> archived
```

## 5.3 旁路状态

```text
proposed -> rejected
planned  -> cancelled
active   -> on_hold
on_hold  -> active
active   -> cancelled
completed -> maintenance
maintenance -> archived
```

## 5.4 生命周期状态的动作控制

这个很关键。状态不只是展示，还要限制动作。

| lifecycle_status | 允许动作 |
|---|---|
| `idea` | 补充描述、提升为 proposed |
| `proposed` | 立项、拒绝、补充调研 |
| `planned` | 加任务、排里程碑、进入 active |
| `active` | 推进任务、启动 workflow、记录决策 |
| `on_hold` | 恢复、取消、补充阻塞原因 |
| `completed` | 复盘、进入维护、归档 |
| `maintenance` | 创建维护任务、归档 |
| `archived` | 只读查询、恢复为 active/maintenance 需显式操作 |
| `cancelled` | 只读、可复盘 |
| `rejected` | 只读、可重新 proposed |

# 6. 当前阶段 `phase`

`lifecycle_status` 是大生命周期，`phase` 是当前具体阶段。

建议第一版 phase：

```text
intake
discovery
planning
design
implementation
validation
review
delivery
maintenance
closure
```

## 6.1 状态解释

| phase | 含义 |
|---|---|
| `intake` | 收集需求、明确项目对象 |
| `discovery` | 探索现状、查资料、找边界 |
| `planning` | 拆任务、定里程碑、定方案 |
| `design` | 方案设计、模型设计、架构设计 |
| `implementation` | 编码、写文档、落地 |
| `validation` | 测试、校验、试运行 |
| `review` | 人工或 agent 审核 |
| `delivery` | 交付、注册、发布、同步 |
| `maintenance` | 后续维护 |
| `closure` | 复盘、归档 |

例如当前这个项目：

```json
"lifecycle_status": "planned",
"phase": "design"
```

等我们开始写文件/CLI 后：

```json
"lifecycle_status": "active",
"phase": "implementation"
```

# 7. 活动状态 `activity_status`

这个表示项目当前“卡在哪里”或者“谁在等谁”。

建议状态：

```text
idle
active
waiting_user
waiting_agent
waiting_external
waiting_review
blocked
scheduled
```

## 7.1 状态解释

| activity_status | 含义 |
|---|---|
| `idle` | 当前没有活跃推进 |
| `active` | 正在推进 |
| `waiting_user` | 等用户确认、补充信息或决策 |
| `waiting_agent` | 等 agent 子任务、后台任务、workflow |
| `waiting_external` | 等外部系统、API、下载、人工第三方 |
| `waiting_review` | 等审核 |
| `blocked` | 有明确阻塞，无法继续 |
| `scheduled` | 已排期，但未开始 |

这个字段是 Project Manager 查询里最有用的。

例如：

```powershell
mycli project-manager query --activity waiting_user
mycli project-manager query --activity blocked
```

可以快速看到哪些项目需要千束处理。

---

# 8. 健康状态 `health_status`

生命周期告诉我们项目在哪，活动状态告诉我们当前卡在哪里，健康状态告诉我们项目是否正常。

建议：

```text
green
yellow
red
gray
```

## 8.1 状态定义

| health_status | 含义 |
|---|---|
| `green` | 正常推进，无明显风险 |
| `yellow` | 有风险或不确定，但还能推进 |
| `red` | 明确阻塞或严重偏离 |
| `gray` | 信息不足，无法判断 |

## 8.2 自动判断建议

第一版可以人工维护，但设计时预留自动计算。

例如：

```text
red 条件：
- activity_status = blocked
- 存在 high severity blocker
- active task 超过 N 天未更新
- 验证失败且没有下一步

yellow 条件：
- 存在 medium risk
- waiting_user 超过 N 天
- scope 未确认但已进入 implementation
- 有任务无 owner / 无 acceptance criteria

gray 条件：
- 没有 status.md
- 没有 active task
- 最近状态缺少 summary
```

# 9. 交付状态 `delivery_status`

这个表示项目成果的完整度。

建议：

```text
none
draft
partial
usable
delivered
verified
published
deprecated
```

## 9.1 状态定义

| delivery_status | 含义 |
|---|---|
| `none` | 尚无交付物 |
| `draft` | 有草稿 |
| `partial` | 有部分成果，但不完整 |
| `usable` | 已能内部使用 |
| `delivered` | 已完成交付 |
| `verified` | 已验证通过 |
| `published` | 已发布/注册/接入正式入口 |
| `deprecated` | 成果已废弃 |

对于 `capability-library` 里的能力包，`published` 可以表示：

- 已写 `SKILL.md`
- 已注册到 skill-library
- 已注册到 mycli
- 可通过 `mycli` 使用

例如 `agent-workflow` 当前大概是：

```json
"delivery_status": "published"
```

当前 `project-manager` 还没文件，所以是：

```json
"delivery_status": "none"
```

# 10. 关注状态 `attention_status`

这个字段用于回答：“现在是否需要人看？”

建议：

```text
none
needs_user_decision
needs_user_input
needs_review
needs_unblock
needs_cleanup
needs_handoff
```

可以允许多个值：

```json
"attention": [
  "needs_user_decision",
  "needs_review"
]
```

## 10.1 常见 attention

| attention | 含义 |
|---|---|
| `needs_user_decision` | 需要用户拍板 |
| `needs_user_input` | 需要用户补充材料 |
| `needs_review` | 需要审核 |
| `needs_unblock` | 需要处理阻塞 |
| `needs_cleanup` | 需要清理状态、文件、任务 |
| `needs_handoff` | 需要生成交接摘要 |
| `needs_scope_confirmation` | 需要确认范围 |
| `needs_validation` | 需要测试/验证 |

这个比单纯 `blocked` 更细。

# 11. 推荐 Project Status 对象

我建议在 `project.json` 里有一个明确的 `status` 对象：

```json
{
  "status": {
    "registration": "registered",
    "lifecycle": "active",
    "phase": "design",
    "activity": "active",
    "health": "green",
    "delivery": "draft",
    "attention": [],
    "updated_at": "2026-04-24T00:00:00+08:00",
    "updated_by": "assistant",
    "summary": "Designing the project status model for project-manager."
  }
}
```

不要把所有状态散在顶层。

# 12. 当前推进项 `current`

项目状态必须绑定“当前推进项”，否则状态没有行动意义。

建议增加：

```json
{
  "current": {
    "goal_id": "goal-0001",
    "milestone_id": null,
    "task_id": "task-0001",
    "next_action_id": "next-0001",
    "owner": "assistant",
    "since": "2026-04-24T00:00:00+08:00",
    "summary": "Draft the project status model and discuss it with user."
  }
}
```

## 12.1 current 的原则

1. 一个项目最多一个 `current.task_id`。
2. 可以有多个 `next_actions`，但只有一个当前推进项。
3. 如果没有当前推进项，`activity_status` 应该是 `idle` 或 `waiting_user`。
4. 如果当前推进项被阻塞，项目 `activity_status` 可以自动变成 `blocked`。
5. current 不是 backlog，它是“现在正在推动什么”。

---

# 13. 推进项 `next_actions`

你说的“项目需要的推进项”，我建议不要直接叫 todo，而叫：

```text
next_actions
```

因为它们不一定都是任务，可能是：

- 等用户确认
- 调研一个问题
- 启动 workflow
- 写某个文件
- 跑测试
- 做一次 review
- 归档项目
- 生成 handoff

## 13.1 next action 数据结构

```json
{
  "id": "next-0001",
  "title": "Confirm project status dimensions",
  "type": "decision",
  "status": "open",
  "priority": "high",
  "owner": "千束",
  "reason": "The status model should be agreed before writing schemas and CLI.",
  "blocks": [
    "task-0002"
  ],
  "created_at": "2026-04-24T00:00:00+08:00",
  "due_at": null
}
```

## 13.2 next action 类型

```text
clarify
decide
design
implement
validate
review
handoff
cleanup
wait
run_workflow
run_agent
```

## 13.3 next action 状态

```text
open
active
waiting
done
cancelled
superseded
```

这个比 task 更轻。  
我的建议是：

- `task` 是正式工作单元。
- `next_action` 是项目当前推进提示。
- 一个 task 可以拆出多个 next actions。
- Project Manager 查询首页优先显示 next actions。

# 14. Project Status 和 Task Status 的关系

项目状态不要被任务状态完全替代。

例如：

```text
Project:
  lifecycle = active
  phase = implementation
  activity = blocked
  health = red

Current Task:
  status = blocked
  blocked_by = user_decision
```

任务状态影响项目状态，但不是同一个东西。

## 14.1 推荐 Task 状态

虽然我们现在先设计项目状态，但为了衔接，我建议任务状态先定成：

```text
proposed
accepted
ready
in_progress
blocked
review
done
cancelled
deferred
```

项目的 `current.task_id` 指向其中一个任务。

# 15. 项目状态机

## 15.1 生命周期状态机

```text
idea
  -> proposed
  -> planned
  -> active
  -> completed
  -> maintenance
  -> archived

proposed -> rejected
planned  -> cancelled
active   -> on_hold
on_hold  -> active
active   -> cancelled
completed -> active       // reopen
maintenance -> active     // major new work
archived -> maintenance   // restore for minor maintenance
archived -> active        // restore for major continuation
```

## 15.2 活动状态机

```text
idle -> active
active -> waiting_user
active -> waiting_agent
active -> waiting_external
active -> waiting_review
active -> blocked
active -> idle

waiting_user -> active
waiting_agent -> active
waiting_external -> active
waiting_review -> active

blocked -> active
blocked -> on_hold
```

## 15.3 健康状态变化

健康状态可以手动设置，也可以根据规则建议：

```text
green -> yellow -> red
red -> yellow -> green
gray -> yellow/green/red
```

但是不要让 health 成为强制状态机，它更像评估结果。

# 16. 项目状态查询视图

为了支持“项目查询”，状态模型应该直接支持这些查询。

## 16.1 常用查询

```powershell
mycli project-manager query --type capability
mycli project-manager query --lifecycle active
mycli project-manager query --activity blocked
mycli project-manager query --attention needs_user_decision
mycli project-manager query --health red
mycli project-manager query --phase design
mycli project-manager query --delivery draft
mycli project-manager query --stale-days 7
```

## 16.2 默认 dashboard

默认项目列表不应该只显示名字，应该显示：

```text
id | name | type | lifecycle | phase | activity | health | current_task | next_action | updated_at
```

例如：

```text
project-manager | Project Manager | capability | active | design | active | green | task-0001 | confirm status model | 2026-04-24
agent-workflow  | Agent Workflow  | capability | maintenance | delivery | idle | green | - | - | 2026-04-23
```

# 17. 状态更新时间和陈旧判断

项目状态必须有 `updated_at`。

建议：

```json
{
  "status": {
    "updated_at": "2026-04-24T00:00:00+08:00",
    "updated_by": "assistant",
    "stale_after_days": 14
  }
}
```

## 17.1 stale 规则

如果：

- `lifecycle` 是 `active` / `planned`
- 且 `updated_at` 超过 `stale_after_days`
- 且没有 active workflow / agent run

则查询时可以标成：

```json
"computed": {
  "is_stale": true
}
```

注意：  
我建议不要直接把 `registration_status` 改成 `stale`，而是在查询视图里计算 `is_stale`。除非索引确实失效。

---

# 18. 风险、阻塞和状态的关系

项目状态里不要塞完整风险列表，只保留摘要。

```json
{
  "status": {
    "health": "yellow",
    "attention": ["needs_user_decision"],
    "blocker_count": 0,
    "risk_count": 2,
    "summary": "Design is progressing, but storage location and lifecycle fields need confirmation."
  }
}
```

完整风险放：

```text
.agent-project/risks.json
```

阻塞项放：

```text
.agent-project/blockers.json
```

或者统一放 task 里。

# 23. 我倾向的最终判断

我建议我们把“项目状态”定义成：

> 项目状态 = 生命周期位置 + 当前阶段 + 当前活动阻塞情况 + 健康度 + 交付成熟度 + 需要关注的事项。

也就是这个核心结构：

```json
{
  "lifecycle": "active",
  "phase": "implementation",
  "activity": "waiting_review",
  "health": "yellow",
  "delivery": "partial",
  "attention": ["needs_review"]
}
```

这个结构足够支持：

- 项目注册
- 项目查询
- 当前状态管理
- 持续推进项
- 多项目 dashboard
- 后续 agent 自动恢复上下文
- 后续 workflow 关联

同时不会一开始就把系统做成 Jira。

# 24. 需要你拍板的点

# 外部调研对项目状态设计的启发

## 1. 状态最好分层，而不是单字段

调研里看到几个典型模型：

- `taskmd`：任务状态  
  `pending | in-progress | completed | in-review | blocked | cancelled`
- ADR：决策状态  
  `proposed | accepted | deprecated | superseded`
- Flow CLI `.STATUS`：项目状态  
  `active | ready | paused | blocked`
- RepoStatus.org：项目成熟度/活跃度  
  `concept | prototypical | functional | production | stable | inactive | suspended`
- GitHub/Nx 等：通过 `tags`、`type`、`projectType`、`status field` 查询

这些都说明一个问题：

> 不同 status 其实回答的是不同问题，不能压成一个字段。

所以我们之前拆成：

```text
lifecycle
phase
activity
health
delivery
attention[]
```

方向是对的。

---

## 2. 可以吸收 RepoStatus 的“成熟度”思想

RepoStatus.org 的状态更像“项目成熟度/维护状态”：

```text
concept
prototypical
functional
production
stable
inactive
suspended
```

这和我刚才提的 `delivery_status` 很接近，但它更偏项目成熟度。

所以这里有两个选择。

### 方案 A：保持 `delivery_status`

```text
none
draft
partial
usable
delivered
verified
published
deprecated
```

优点：更适合 capability-library，因为“是否注册到 mycli/skill-library”很重要。

### 方案 B：改名成 `maturity_status`

```text
concept
prototype
functional
usable
stable
production
deprecated
```

优点：更通用，借鉴 RepoStatus。

### 我的建议

保留 `delivery_status`，但把部分枚举调整得更像成熟度：

```text
none
concept
draft
prototype
usable
delivered
verified
published
deprecated
```

这样兼顾：

- 早期概念：`concept`
- 初稿：`draft`
- 原型：`prototype`
- 可用：`usable`
- 已交付：`delivered`
- 已验证：`verified`
- 已发布/注册：`published`

对于能力图书馆，`published` 很关键。

---

## 3. 项目状态可以有 `.STATUS`/`status.md` 双层表达

调研里 Flow CLI `.STATUS` 很有启发：

```text
project: mediationverse
type: r-package
status: active
priority: P0
progress: 85
next: Run final simulations
updated: 2025-12-24
```

这个非常适合 agent 快速读。

但我们需要更结构化，所以可以设计两层：

```text
.agent-project/project.json     机器事实来源
.agent-project/status.md        人类可读状态页
.agent-project/STATUS           可选快速状态摘要
```

我倾向第一版只做：

```text
project.json
status.md
```

但 `status.md` 顶部可以放 YAML frontmatter：

```md
---
lifecycle: planned
phase: design
activity: waiting_user
health: green
delivery: draft
updated_at: 2026-04-24T00:00:00+08:00
---

# Project Status

...
```

不过如果已经有 `project.json`，`status.md` frontmatter 是否重复要谨慎。我的建议：

- `project.json` 是唯一结构化事实来源。
- `status.md` 是由 project-manager 生成或手工维护的人类摘要。
- 不在 `status.md` 里重复所有字段，最多放简短摘要。

---

## 4. 状态应该支持查询和 dashboard

调研里 Nx、GitHub Projects、taskmd 都强调：

- status
- tags
- priority
- phase
- updated time
- dependencies
- filters

所以项目状态设计应天然支持：

```powershell
query --type capability
query --lifecycle active
query --activity blocked
query --attention needs_user_decision
query --health red
query --delivery published
query --stale-days 14
```

这说明我们的 status 对象里一定要有：

```json
"type": "capability",
"tags": [],
"status": {
  "lifecycle": "...",
  "phase": "...",
  "activity": "...",
  "health": "...",
  "delivery": "...",
  "attention": []
},
"timestamps": {
  "updated_at": "...",
  "last_activity_at": "..."
}
```

---

## 5. 外部模型支持“一个概念一个文件”

调研里 taskmd 和 ADR 都有一个强模式：

> 一个任务一个文件，一个决策一个文件。

这对后面的任务和决策设计很重要，但对“项目状态”也有影响：

- 项目总状态放 `project.json`
- 当前人类摘要放 `status.md`
- 任务不要都塞进 `project.json`
- 决策不要塞进 `project.json`
- `project.json` 只保存引用和汇总

所以项目状态里只放：

```json
"current": {
  "task_id": "task-0001",
  "next_action_id": "next-0001"
}
```

不把完整 task 内容塞进去。

# 我建议的状态机核心

## lifecycle 主路径

```text
idea -> proposed -> planned -> active -> completed -> maintenance -> archived
```

旁路：

```text
proposed -> rejected
planned -> cancelled
active -> on_hold
on_hold -> active
active -> cancelled
completed -> active
maintenance -> active
archived -> active
```

## activity 主路径

```text
idle -> active
active -> waiting_user
active -> waiting_agent
active -> waiting_external
active -> waiting_review
active -> blocked
active -> idle

waiting_* -> active
blocked -> active
blocked -> on_hold
```

## delivery 主路径

```text
none -> concept -> draft -> prototype -> usable -> delivered -> verified -> published
published -> deprecated
```

不同项目类型可以跳过某些状态。

例如文档项目可能：

```text
none -> draft -> delivered -> published
```

能力包可能：

```text
none -> draft -> usable -> verified -> published
```

研究项目可能：

```text
none -> draft -> delivered
```

# 结论

调研后我更确信，项目状态应采用这个最终结构：

```json
"status": {
  "registration": "registered",
  "lifecycle": "active",
  "phase": "implementation",
  "activity": "waiting_review",
  "health": "yellow",
  "delivery": "usable",
  "attention": ["needs_review"],
  "summary": "...",
  "updated_at": "...",
  "updated_by": "..."
}
```

它的优点是：

- 比单字段状态清楚。
- 支持多项目查询。
- 支持当前推进项。
- 能表达项目生命周期、阻塞、健康度和交付成熟度。
- 能和任务、决策、workflow run 解耦。
- 适合文件系统持久化。
- 适合 agent 恢复上下文。

我的建议是：**先把这套状态模型定为 project-manager v0.1 的核心模型。**  
下一步再设计：

1. `project.json` 完整 schema
2. `status.md` 模板
3. 项目 registry/index
4. `next_actions` 和 task 模型
5. query 命令返回格式
