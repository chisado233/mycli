## 3. 需求拆解

### 3.1 业务/产品能力

本次是运营性巡检，不是产品开发。需要的能力主要是：

- 识别当前是否有待处理任务
- 区分 `draft` / `listed` / `claimed`
- 判断是否需要进入“等待调度”状态

### 3.2 技术/工程能力

所需能力：

- 能读取 `mycli` 包定义和帮助信息
- 能通过 `mycli task-hall` 获取当前任务大厅状态
- 能根据协议判断是否应该 claim、等待、或结束生命周期

### 3.3 数据/接口/集成能力

涉及的集成点：

- `mycli task-hall`
- `mycli project-manager`
- `agent-cli` / schedule 相关机制仅在“等待后续事件”时才需要介入

### 3.4 测试/验证/质量能力

本次验证标准很明确：

- 能确认是否存在 `listed` 任务
- 能确认是否存在 `claimed` 任务
- 能确认 task-hall 是否仅承担生命周期管理而非调度
- 能给出下一步动作建议

### 3.5 部署/运维/监控能力

本次不涉及部署，但涉及轻量运维判断：

- 是否进入等待状态
- 是否需要创建 schedule 维持会话生命周期

### 3.6 文档/协作/运营能力

需要：

- 基于现有协议文档输出清晰状态总结
- 在没有合适任务时，不误 claim，不伪造活跃工作
