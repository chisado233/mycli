### 2.2 发现的现有实现与模式

#### 工作区结构模式

该 workspace 不是单一 repo，而是一个以 `capability-library/` 为核心的 agent 工作区：

#### mycli 设计模式

`mycli` 的核心约定很明确：

- 目录即 package 树
- `cli.package.json` 是命令注册中心
- `README.md` 是 help 来源
- `common/cli.ps1` 是运行时核心
- `entry` 使用绝对路径
- `prefixArgs` 用于原生命令包装/转发

这意味着后续若要新增能力、改注册、扩任务流，首选沿用 `mycli` 的 package/command 模型，而不是临时散脚本。

#### task-hall 模式

`task-hall` 已经形成相对完整的协议：

- 任务入口：JSON request
- 任务主体：Markdown `task.md`
- 领取：claim request JSON
- 审核：report markdown + reviewer decision
- 状态：`draft -> listed -> claimed -> done/cancelled/archived`

关键点：

- `task.md` 必须有 `## 领取门槛`
- skill 暂不作为第一版硬性 claim 条件
- 前台 agent 与 reviewer agent 都通过 `agent-cli` / opencode 触发
- 审核是文本一致性审核，不是实际执行验证

## 6. 缺口与新增 skill 建议

当前最明显的缺口不是代码实现能力，而是 **操作制度层的固化能力**。

### 缺口 1：Heartbeat 标准流程未显式封装

用户要求里提到：

- inspect current context and task-hall
- no active work 时寻找 suitable listed task 并先 fully read
- waiting after publishing/delegating work 时创建 agent-cli schedule
- lifecycle should end 时输出 `AGENT_CLI_LIFECYCLE_END`

但目前我看到的是：

- `task-hall` 有
- `project-manager` 有
- `agent-cli` 有
- 但“heartbeat 整体 SOP”没有看到现成统一 skill / 命令封装

**建议新增：**

- `heartbeat-session-manager` skill
- 或在 `agent-cli` 下新增一组 session lifecycle/schedule SOP 文档和命令包装

### 缺口 2：active work 判定标准未完全外显

目前能查：

- `task-hall` listed/claimed 状态
- `project-manager` list/status/current

但“什么叫当前 session 有 active work”还未见统一判定文档。

**建议新增：**

- 一份 “active work 判定矩阵”
  - task-hall claimed 算 active
  - project-manager 当前 task/next action 处于 in_progress 算 active
  - 仅有 draft 不一定算 active
  - 仅有 done 不算 active

### 缺口 3：等待态 schedule 机制未显式确认

用户要求“waiting after publishing or delegating work, create an agent-cli schedule for this session”，但当前未核实到现成命令与规范。

**建议：**

- 先核实 `mycli agent-cli` 是否已有 schedule / resume / session-watch 能力
- 若无，新增轻量 SOP 或命令包装

### 缺口 4：task-hall 与 project-manager 的切换规则不清晰

两者分别处理：

- one-off 任务
- durable 项目

但何时从 task-hall 进入 project-manager，缺少明确约定。

**建议：**

- 增加一份“任务 vs 项目”的升级/降级判断规则

## 8. 风险、前置条件与需要用户确认的问题

### 风险

- **风险 1：误把无 listed task 当成无工作**
  - 事实上还可能存在 durable project、外部委派、session 内待续动作
- **风险 2：过度自动化**
  - 在未确认 schedule 机制前，直接假设并执行“会话计划”可能越界
- **风险 3：task-hall 与 project-manager 语义混淆**
  - 会导致后续 claim / project tracking 流程混乱
- **风险 4：skill 库里已有能力但未统一封装**
  - 若不先盘点，会重复造轮子

### 前置条件

若要继续把 heartbeat 做成稳定工作流，需要先确认：

1. `agent-cli` 是否已经支持 session schedule / delayed resume
2. 当前是否存在一个应被视为 active 的 durable project
3. 你希望 heartbeat 更偏：
   - “人工巡检 SOP”
   - 还是“半自动调度机制”

### 需要用户确认的问题

1. 这次 heartbeat 后，你希望我继续做哪一类后续工作？

   - C. 检查 `project-manager` 中是否存在当前 session 应继续的 active project

2. “If no active work” 中的 active work，你希望采用哪种判定口径？
   - 仅 task-hall claimed

3. 是否要把本次分析沉淀为一个正式 skill / 文档草案？
