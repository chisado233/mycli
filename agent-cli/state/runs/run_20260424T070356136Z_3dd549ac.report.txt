千束，我建议 `project-manager` 的结构要分成两层：

1. **项目内结构**：每个项目自己的 `.agent-project/`，负责长期事实来源。
2. **全局注册/索引结构**：project-manager 自己维护一个 registry/index，负责跨项目查询。

也就是说，不要只靠扫描每个项目的 README；也不要只靠全局 registry。  
应该是：

```text
项目内 .agent-project/ = 单项目真相来源
全局 registry/index   = 查询加速与统一入口
```

这样才能支持你说的：

- 按状态查
- 按名称查
- 按生命周期查
- 按项目类型查
- 按持续/有限项目查
- 按当前阶段查
- 按健康度查
- 按是否需要用户处理查
- 按更新时间/陈旧状态查
- 按标签/领域/owner 查

# 1. 设计参考

我刚查了本地和外部模式，几个结论很明确。

## 1.1 本地已有旧模式：`project-skill-manager`

旧的 `project-skill-manager` 是通过扫描：

```text
D:\agent_workspace\projects\*/README.md
```

读取 README YAML frontmatter：

```yaml
---
name: 项目名称
description: 一句话描述项目
status: todo/done/sustained
---
```

脚本返回：

```json
{
  "name": "项目名称",
  "status": "状态",
  "description": "描述",
  "path": "项目路径"
}
```

这个模式的优点：

- 简单
- 人类可编辑
- 可以按 status 过滤
- 适合早期

缺点也明显：

- status 太粗
- 查询字段太少
- 无生命周期、阶段、健康度、持续运行信息
- README 既做人类文档又做机器事实来源，容易混乱
- 不适合复杂查询和持续项目

所以新设计应该继承它的“可扫描、可注册、可查询”思想，但升级为结构化 `.agent-project/project.json`。

---

## 1.2 本地 `skill-library` registry 模式

`mycli skill-library` 当前有：

```text
capability-library/mycli/skill-library/registry.json
```

它通过扫描 `SKILL.md` frontmatter 生成 registry，registry 用于快速查询。

这个模式对 project-manager 很有价值：

```text
source of truth: 每个 skill 的 SKILL.md
query index: mycli/skill-library/registry.json
```

迁移到项目管理就是：

```text
source of truth: 每个项目的 .agent-project/project.json
query index: project-manager/registry.json 或 index.json
```

`agent-workflow` 已经有一个很成熟的工程结构：

```text

    manifest.json
    graph.json
```

## 1.4 外部模式：Backstage / Kubernetes / Nx

几个外部系统对查询很有启发。

### Backstage

Backstage 的 `catalog-info.yaml` 结构是：

```yaml
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: artist-web
  description: ...
  labels:
    example.com/custom: value
  annotations:
    example.com/service-discovery: artistweb
  tags:
    - java
spec:
  type: website
  lifecycle: production
  owner: team
  system: public-websites
```

核心启发：

```text
metadata = 名称、描述、标签、链接、注解
spec     = 类型、生命周期、owner、system
```

非常适合我们借鉴。

### Kubernetes

Kubernetes 最强的是 labels/selectors：

```yaml
labels:
  app.kubernetes.io/name: wordpress
  app.kubernetes.io/instance: wordpress-abcxyz
  app.kubernetes.io/component: server
  app.kubernetes.io/part-of: wordpress
  app.kubernetes.io/managed-by: Helm
```

核心启发：

- label 用于查询和分组
- annotation 用于外部系统/说明性元数据
- selector 支持多个条件 AND 查询

### Nx

Nx `project.json` 支持：

```json
{
  "name": "mylib",
  "root": "libs/mylib",
  "projectType": "library",
  "tags": ["scope:myteam"],
  "metadata": {},
  "targets": {}
}
```

核心启发：

- 每个项目有 `root`
- 项目有 `projectType`
- tags 用于图和查询
- target 表示可执行动作

对我们来说，`targets` 可以演化成“项目可执行动作/常用命令”，但 MVP 先不必重。

# 4. `project.json` 结构设计

我建议采用类似 Backstage 的 envelope：

```json
{
  "apiVersion": "agent-project/v1",
  "kind": "Project",
  "metadata": {},
  "spec": {},
  "status": {},
  "current": {},
  "operations": {},
  "relations": {},
  "timestamps": {}
}
```

为什么这样分？

| 字段 | 作用 |
|---|---|
| `apiVersion` | schema 版本，方便以后迁移 |
| `kind` | 未来可能有 ProjectGroup / Program |
| `metadata` | 用于发现、展示、查询 |
| `spec` | 项目本质定义，不频繁变化 |
| `status` | 当前状态，经常变化 |
| `current` | 当前推进项 |
| `operations` | 持续运行项目运行状态 |
| `relations` | 和 workflow、skill、repo、项目之间的关系 |
| `timestamps` | 创建、更新、索引时间 |

# 6. 为什么要有 `metadata.labels`

`tags` 是人类友好的：

```json
"tags": ["daily", "news", "automation"]
```

`labels` 是机器查询友好的：

```json
"labels": {
  "agent.project/type": "research",
  "agent.project/mode": "ongoing",
  "agent.project/domain": "content-ops"
}
```

这样未来可以支持类似 Kubernetes 的 selector：

```powershell
mycli project-manager query --selector "agent.project/mode=ongoing,agent.project/domain=content-ops"
```

第一版可以不实现完整 selector，但结构先留好。

# 9. 查询模式设计

你提到“基于状态，基于名称，基于生命周期什么的查询”，我建议 query 命令支持两层。

## 9.1 简单参数查询

```powershell
mycli project-manager query
mycli project-manager query --name news
mycli project-manager query --id daily-hot-news
mycli project-manager query --type research
mycli project-manager query --mode ongoing
mycli project-manager query --lifecycle active
mycli project-manager query --phase operations
mycli project-manager query --activity stable
mycli project-manager query --health yellow
mycli project-manager query --delivery published
mycli project-manager query --attention needs_maintenance
mycli project-manager query --owner 千束
mycli project-manager query --tag automation
mycli project-manager query --domain content-ops
mycli project-manager query --stale
```

支持组合：

```powershell
mycli project-manager query --mode ongoing --health yellow
mycli project-manager query --type capability --lifecycle active
mycli project-manager query --phase maintenance --attention needs_unblock
```

默认 AND 语义。

---

## 9.2 Selector 查询

未来支持：

```powershell
mycli project-manager query --selector "agent.project/mode=ongoing,agent.project/domain=content-ops"
```

或者：

```powershell
mycli project-manager query --where "status.lifecycle=active,status.health!=green"
```

但第一版不建议上复杂表达式。

# 14. 项目注册流程

## 14.1 init

```powershell
mycli project-manager init <project-root> --id <id> --name <name> --type <type> --mode finite|ongoing
```

生成：

```text
.agent-project/
  project.json
  status.md
  tasks/
  next-actions/
  decisions/
  context/
  operations/
  runs/
  handoffs/
  .generated/
```

## 14.2 register

```powershell
mycli project-manager register <project-root>
```

逻辑：

1. 检查 `.agent-project/project.json`
2. 读取 metadata/spec/status
3. 写入/更新全局 `registry.json`
4. 生成摘要字段
5. 标记 `last_indexed_at`

## 14.3 scan

```powershell
mycli project-manager scan D:\agent_workspace\projects
```

扫描：

```text
**/.agent-project/project.json
```

可兼容旧模式：

```text
README.md YAML frontmatter
```

如果发现旧格式：

```yaml
status: sustained
```

映射为：

```json
"project_mode": "ongoing",
"lifecycle": "active",
"phase": "operations"
```

# 20. 我建议拍板的核心设计

我建议 project 结构定成：

```text
.agent-project/project.json 是单项目事实来源
project-manager/registry.json 是全局查询索引
status.md 是人类可读摘要
tasks/ next-actions/ decisions/ 分文件存储
operations/ 专门支持持续项目
relations 支持项目/skill/workflow/repo 关联
labels/tags 支持灵活查询
```

核心 `project.json` 采用：

```json
{
  "apiVersion": "agent-project/v1",
  "kind": "Project",
  "metadata": {},
  "spec": {},
  "status": {},
  "current": {},
  "operations": {},
  "relations": {},
  "timestamps": {}
}
```

核心查询字段压平到 registry：

```text

project_mode
owner
domain

labels
lifecycle
phase
activity
health
delivery
attention
current_task_id
```

这样查询会很舒服：

```powershell

mycli project-manager query --type capability --delivery published
mycli project-manager query --attention needs_user_decision
mycli project-manager query --name 素材库
```

这套结构既兼容旧的 `todo/done/sustained` 思路，也能支持持续性项目和未来复杂查询。千束，librarian 的调研也回来了。它进一步验证了：**project-manager 的项目结构最好采用“分布式项目事实源 + 中央 registry/index”的模式**，并且字段设计应该借鉴 Backstage / Kubernetes / Nx 这类“可查询目录系统”。

我把之前方案再收敛成一个更明确的 **Project Structure v0.4**。

# Project Structure v0.4

## 0. 总体判断

我建议采用：

```text

  .agent-project/project.json 作为项目事实来源

  registry.json 作为项目注册表
```

| Kubernetes | 对象自带 metadata/labels，API server 支持 selector 查询 |
| project-manager | 每个项目有 `.agent-project/project.json`，全局 registry 支持查询 |

# 2. `project.json` 顶层结构

我建议借鉴 Backstage 的 envelope 模型：

```json
{
  "apiVersion": "agent-project/v1",
  "kind": "Project",
  "metadata": {},
  "spec": {},
  "status": {},
  "current": {},
  "operations": {},
  "relations": {},
  "timestamps": {}
}
```

## 为什么这样分？

| 区块 | 职责 | 更新频率 |
|---|---|---|
| `apiVersion` | schema 版本 | 很少 |
| `kind` | 对象类型，未来可扩展 ProjectGroup | 很少 |
| `metadata` | 名称、描述、标签、owner、查询元数据 | 中低 |
| `spec` | 项目本质定义，目标、范围、模式 | 中低 |
| `status` | 当前状态 | 高频 |
| `current` | 当前推进项 | 高频 |
| `operations` | 持续运行信息 | 高频/自动 |
| `relations` | 和项目、workflow、skill、repo 的关系 | 中 |
| `timestamps` | 时间戳 | 高频 |

这个结构的好处是：

- 机器好解析；
- 人也容易理解；
- 查询字段可以从 `metadata/spec/status/current/operations` 中压平到 registry；
- 后续 schema 迁移容易。

# 3. `metadata` 设计

`metadata` 主要服务于 **发现、展示、查询、归类**。

```json
"metadata": {
  "id": "daily-hot-news",
  "name": "每日新闻热点收集",
  "aliases": ["热点收集", "daily-news"],
  "description": "每天自动收集和整理热点新闻主题。",
  "root": "D:\\agent_workspace\\projects\\daily-hot-news",
  "owner": "千束",
  "domain": "content-ops",
  "tags": ["daily", "news", "hot-topics", "automation"],
  "labels": {
    "agent.project/type": "research",
    "agent.project/mode": "ongoing",
    "agent.project/domain": "content-ops",
    "agent.project/managed-by": "project-manager"
  },
  "annotations": {
    "notes": "持续运行型内容运营项目"
  },
  "links": [
    {
      "title": "Project README",
      "url": "README.md",
      "type": "docs"
    }
  ]
}
```

## 3.1 `id`

必须稳定。

建议规则：

```text
小写 kebab-case
不要用中文作为 id
不要频繁改
```

示例：

```text
project-manager
daily-hot-news
asset-library
agent-workflow
```

## 3.2 `name`

人类可读，可以中文：

```json
"name": "每日新闻热点收集"
```

## 3.3 `aliases`

用于名称查询。

比如用户说：

```text
热点项目
每日新闻
daily-news
```

都可以匹配到同一项目。

## 3.4 `tags` vs `labels` vs `annotations`

这三者要区分清楚。

| 字段 | 类型 | 用途 |
|---|---|---|
| `tags` | string[] | 人类友好的松散标签 |
| `labels` | key/value | 机器查询、selector、分类 |
| `annotations` | key/value | 外部系统引用、备注、非筛选主字段 |

借鉴 Kubernetes / Backstage：

```json
"tags": ["automation", "news"]
```

用于简单查询：

```powershell
query --tag automation
```

```json
"labels": {
  "agent.project/mode": "ongoing"
}
```

用于结构化查询：

```powershell
query --selector "agent.project/mode=ongoing"
```

```json
"annotations": {
  "github.com/project-slug": "owner/repo"
}
```

用于外部系统引用，不一定参与核心筛选。

# 6. `current` 设计

`current` 是项目首页最有用的部分，回答：

```text
现在正在推进什么？
下一步是什么？
卡在哪里？
```

```json
"current": {
  "goal_id": "goal-0001",
  "milestone_id": null,
  "task_id": "task-0001",
  "next_action_id": "next-0001",
  "summary": "确认项目结构 v0.4，并准备 project.json schema。",
  "owner": "assistant",
  "since": "2026-04-24T09:00:00+08:00"
}
```

原则：

1. 一个项目最多一个 `current.task_id`。
2. 可以有多个 next-actions，但 `current.next_action_id` 指向当前最重要的。
3. 如果没有当前推进项，`activity` 应该是 `stable` / `queued` / `waiting`，而不是假装 `in_progress`。
4. `current` 只存引用和摘要，不存完整任务正文。

# 12. 查询模式设计

## 12.1 基础查询

```powershell
mycli project-manager query
mycli project-manager query --id daily-hot-news
mycli project-manager query --name news
mycli project-manager query --name 热点
```

`--name` 应匹配：

```text
id
name
aliases
description
```

也可以分开：

```powershell
--id
--name
--alias
--text
```

但第一版 `--name` 做模糊匹配即可。

---

## 12.2 类型查询

```powershell
mycli project-manager query --type research
mycli project-manager query --mode ongoing
mycli project-manager query --owner 千束
mycli project-manager query --domain content-ops
mycli project-manager query --priority high
mycli project-manager query --tag automation
```

---

## 12.3 状态查询

```powershell
mycli project-manager query --registration indexed
mycli project-manager query --lifecycle active
mycli project-manager query --phase operations
mycli project-manager query --activity stable
mycli project-manager query --health yellow
mycli project-manager query --delivery published
mycli project-manager query --attention needs_maintenance
```

组合查询默认 AND：

```powershell
mycli project-manager query --mode ongoing --health yellow
mycli project-manager query --lifecycle active --attention needs_user_decision
mycli project-manager query --phase maintenance --activity blocked
```

---

## 12.4 持续运行查询

```powershell
mycli project-manager query --operations-enabled
mycli project-manager query --cadence daily
mycli project-manager query --last-run-status failed
mycli project-manager query --consecutive-failures 3
mycli project-manager query --next-run-before 2026-04-25T00:00:00+08:00
```

第一版可以先支持：

```powershell
--mode ongoing
--last-run-status failed
--health red
--attention needs_operation_check
```

不急着做复杂时间比较。

---

## 12.5 时间/陈旧查询

```powershell
mycli project-manager query --stale
mycli project-manager query --updated-before 2026-04-01
mycli project-manager query --inactive-days 14
```

陈旧判断：

```text
is_stale = now - status.updated_at > spec.stale_after_days
```

持续项目可以额外判断：

```text
now > operations.next_run_at + tolerance
last_run_status failed
consecutive_failures > 0
```

---

## 12.6 selector 查询

未来支持 Kubernetes 风格：

```powershell
mycli project-manager query --selector "agent.project/mode=ongoing,agent.project/domain=content-ops"
```

再高级一点：

```powershell
mycli project-manager query --selector "agent.project/type in (research,documentation),agent.project/mode=ongoing"
```

但 MVP 不要先做复杂 parser，先保留 `labels` 结构即可。

```md
# ADR-0001: Use distributed project.json plus central registry

## Status

Proposed

## Context

Projects need to support query by status, lifecycle, name, type, tags, owner, and ongoing operation state.

## Decision

Each project owns `.agent-project/project.json` as the source of truth.
```

# 17. 旧 project-skill-manager 兼容

旧格式：

```yaml
---
name: 项目名称
description: 一句话描述项目
status: todo/done/sustained/skill_dependent
---
```

映射：

| 旧状态 | 新结构 |
|---|---|
| `todo` | `project_mode=finite`, `lifecycle=active/planned`, `phase=construction`, `activity=in_progress` |
| `done` | `project_mode=finite`, `lifecycle=completed`, `phase=closure`, `activity=stable` |
| `sustained` | `project_mode=ongoing`, `lifecycle=active`, `phase=operations`, `activity=stable` |
| `skill_dependent` | 不作为主状态，转成 `labels/protection` |

例如：

```json
"labels": {
  "agent.project/protection": "skill-dependent"
}
```

或：

```json
"protection": ["skill_dependent", "do_not_delete"]
```

# 19. 我建议 v0.4 拍板内容

我建议锁定以下结构作为 project-manager v0.1 的基础：

```text
项目内：
  .agent-project/project.json
  .agent-project/status.md
  .agent-project/tasks/
  .agent-project/next-actions/
  .agent-project/decisions/
  .agent-project/context/
  .agent-project/operations/
  .agent-project/runs/
  .agent-project/handoffs/

全局：
  project-manager/registry.json
```

`project.json` 顶层：

```json
{
  "apiVersion": "agent-project/v1",
  "kind": "Project",
  "metadata": {},
  "spec": {},
  "status": {},
  "current": {},
  "operations": {},
  "relations": {},
  "timestamps": {}
}
```

查询优先支持：

```text
id/name
type/mode
owner/domain/tag
lifecycle/phase/activity/health/delivery/attention
stale/missing
last-run-status for ongoing
```

命令优先支持：

```powershell
init
register
scan
refresh
list
query
get
status
```

这套结构能覆盖：

- 普通一次性项目；
- 持续运行项目；
- 素材库这种持续积累项目；
- 每日新闻热点这种定时运行项目；
- capability-library 能力建设项目；
- 后续维护；
- 多条件项目查询。
